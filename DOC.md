# Криптографические примитивы

## 1. Электронная подпись PKCS#7

### Назначение

Электронная подпись гарантирует три свойства:

- **Аутентичность** — документ подписан владельцем приватного ключа
- **Целостность** — документ не был изменён после подписания
- **Неотрекаемость** — подписант не может отказаться от факта подписания

### Стандарт PKCS#7 / CMS

PKCS#7 (RFC 2315) и его преемник CMS (RFC 5652) определяют **бинарный формат** контейнера, содержащего:

- Подписанные данные (или только хеш, если подпись отсоединённая)
- Информацию о подписанте (SignerInfo): алгоритм подписи, сертификат, значение подписи
- Информацию о сертификатах (Certificates)

ASN.1 структура подписанного сообщения:

```
CMS SignedData ::= SEQUENCE {
  version                  Version,
  digestAlgorithms         DigestAlgorithmIdentifiers,
  contentInfo              ContentInfo,
  certificates             [0] IMPLICIT CertificateSet OPTIONAL,
  revocationInfoChoice     [1] IMPLICIT RevocationInfoChoices OPTIONAL,
  signerInfos              SignerInfos
}
```

### Алгоритм подписания

```
┌──────────────────────────────────────────────────────────────┐
│                     ПОДПИСАНИЕ                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Digest = SHA-256(Документ)                               │
│     → 32 байта хеш-дайджеста                                 │
│                                                              │
│  2. Signature = RSA_Sign(PrivKey, Digest)                    │
│     → RSA PKCS#1 v1.5 padding                               │
│     → SHA256withRSA алгоритм                                │
│                                                              │
│  3. PKCS#7 = Wrap(Signature, Certificate, Metadata)         │
│     → ASN.1 DER кодирование                                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Отсоединённая vs Присоединённая

| Параметр            | Отсоединённая (Detached) | Присоединённая (Attached)        |
| ------------------- | ------------------------ | -------------------------------- |
| Данные в контейнере | ❌ Нет                   | ✅ Да                            |
| Размер контейнера   | ~2–4 КБ (только подпись) | Размер данных + подпись          |
| Хранение данных     | Отдельно                 | Внутри PKCS#7                    |
| Использование       | Большие файлы, архивы    | Маленькие документы              |
| Верификация         | Требуется исходный файл  | Данные извлекаются из контейнера |

### Алгоритм верификации

```
┌──────────────────────────────────────────────────────────────┐
│                      ВЕРИФИКАЦИЯ                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Parse(PKCS#7) → Signature, SignerCert, Algorithm        │
│     → Извлечь подпись и сертификат подписанта               │
│                                                              │
│  2. PubKey = SignerCert.getPublicKey()                      │
│     → Получить открытый ключ из сертификата                 │
│                                                              │
│  3. ExtractData(PKCS#7) → OriginalData                     │
│     → Извлечь данные из контейнера (attached)               │
│     → Или использовать external data (detached)             │
│                                                              │
│  4. ComputedHash = SHA-256(OriginalData)                    │
│     → Независимо вычислить хеш                              │
│                                                              │
│  5. RecoveredHash = RSA_Verify(PubKey, Signature)           │
│     → Дешифровать подпись открытым ключом                   │
│                                                              │
│  6. Valid = (ComputedHash == RecoveredHash)                │
│     → Совпадение → подпись действительна                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 2. Гибридное шифрование AES + RSA

### Назначение

Обеспечение **конфиденциальности** данных:

- Только владелец приватного ключа может расшифровать данные
- AES-GCM дополнительно гарантирует **целостность** зашифрованных данных

### Почему гибридная схема?

| Алгоритм | Скорость      | Макс. размер данных    | Назначение        |
| -------- | ------------- | ---------------------- | ----------------- |
| RSA      | ~1000 ops/sec | 4096/8 − 11 ≈ 506 байт | Шифрование ключей |
| AES-GCM  | ~10 Gbps      | Без ограничений        | Шифрование данных |

Гибридная схема использует:

- **RSA** для безопасной передачи AES-ключа (key encapsulation)
- **AES-GCM** для шифрования самих данных

### Алгоритм шифрования

```
┌──────────────────────────────────────────────────────────────┐
│                      ШИФРОВАНИЕ                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. AES_KEY = Random(32 bytes)                               │
│     → Криптографически случайный 256-битный ключ            │
│                                                              │
│  2. IV = Random(12 bytes)                                    │
│     → Уникальный nonce для GCM                              │
│                                                              │
│  3. CIPHERTEXT = AES_GCM_Encrypt(AES_KEY, IV, Data)         │
│     → Включает 16-байтный authentication tag                │
│                                                              │
│  4. WRAPPED_KEY = RSA_Encrypt(PublicKey, AES_KEY)           │
│     → RSA PKCS#1 v1.5 padding                               │
│                                                              │
│  5. OUTPUT = Concat(WRAPPED_KEY_LEN, WRAPPED_KEY, IV,       │
│                    CIPHERTEXT)                               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Бинарный формат результата

```
┌──────────────────────────────────────────────────────────────┐
│  Byte  0–3   │ 4–N    │ N+4–N+15 │ N+16–End                │
│  ─────────   │ ────── │ ───────── │ ─────────               │
│  int32 BE    │ RSA    │ IV       │ AES-GCM ciphertext      │
│  (key len)  │ (AES)  │ (12B)    │ (data + 16B tag)        │
└──────────────────────────────────────────────────────────────┘

где N = 4 + len(encrypted_aes_key)
```

### Алгоритм дешифрования

```
┌──────────────────────────────────────────────────────────────┐
│                      ДЕШИФРОВАНИЕ                             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Parse: keyLen, wrappedKey, iv, ciphertext               │
│     → Разобрать структуру                                   │
│                                                              │
│  2. AES_KEY = RSA_Decrypt(PrivateKey, wrappedKey)           │
│     → Дешифровать AES-ключ приватным ключом                 │
│                                                              │
│  3. DATA = AES_GCM_Decrypt(AES_KEY, iv, ciphertext)         │
│     → Проверка authentication tag                           │
│     → При ошибке → данные повреждены                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Безопасность GCM режима

**GCM (Galois/Counter Mode)** — аутентифицированный режим шифрования, обеспечивающий:

- **Конфиденциальность** — данные невидимы без ключа
- **Аутентичность** — невозможно подделать зашифрованные данные
- **Аутентификация дополнительных полей (AAD)** — можно привязать контекст

**Важно:**

- IV **никогда не должен повторяться** для одного ключа
- В проекте IV генерируется криптографически случайным образом для каждой операции
- Размер IV 12 байт — рекомендуемый для GCM

## 3. Хэширование

### SHA-256

**Семейство:** SHA-2 (FIPS 180-4)
**Размер блока:** 512 бит
**Размер вывода:** 256 бит (64 hex символа)
**Количество раундов:** 64

**Внутреннее устройство (упрощённо):**

```
┌──────────────────────────────────────────────────────────────┐
│                    SHA-256 алгоритм                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Padding: довести длину до кратности 512 бит             │
│     → Добавить бит '1', нули, 64-битную длину               │
│                                                              │
│  2. Разбить на блоки по 512 бит                             │
│                                                              │
│  3. Для каждого блока:                                      │
│     a. Message Schedule: W[0..63]                           │
│     b. 64 раунда с рабочими переменными a,b,c,d,e,f,g,h    │
│     c. Модификация хеш-регистра H[0..7]                     │
│                                                              │
│  4. Финальный хеш = H[0..7] (8 × 32 бит = 256 бит)        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**Криптостойкость:**

- **Pre-image resistance:** 2²⁵⁶ операций
- **Second pre-image resistance:** 2²⁵⁶ операций
- **Collision resistance:** 2¹²⁸ операций (birthday attack)

### PBKDF2

**Стандарт:** RFC 2898 / FIPS 186-4
**Назначение:** Преобразование пароля (low entropy) в криптографический ключ (high entropy)

**Алгоритм:**

```
DK = PBKDF2(Password, Salt, c, dkLen):

    hLen = length(HMAC output)  // 32 bytes for SHA-256
    blocks = ceil(dkLen / hLen)

    for i = 1 to blocks:
        U1 = HMAC(Password, Salt || INT(i))
        U2 = HMAC(Password, U1)
        ...
        Uc = HMAC(Password, Uc-1)
        T_i = U1 XOR U2 XOR ... XOR Uc

    DK = T_1 || T_2 || ... || T_blocks[0:dkLen]
```

**Параметры в проекте:**
| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Псевдослучайная функция | HMAC-SHA256 | Стандарт FIPS 186-4 |
| Размер соли | 16 байт | Рекомендуется NIST SP 800-132 |
| Итераций | 10000 | Базовый минимум |
| Размер ключа | 32 байта (256 бит) | Совместимо с AES-256 |

**Рекомендации для продакшена:**

- Увеличить итерации до **≥100000** (или использовать bcrypt/Argon2)
- Никогда не использовать PBKDF2 без соли
- Хранить соль рядом с результатом (не секретна)

## 4. Управление ключами

### Контейнеры ключей

#### JKS (Java KeyStore)

- **Формат:** Проприетарный Oracle
- **Структура:** Бинарный файл с зашифрованными приватными ключами и сертификатами
- **Шифрование ключей:** 3DES/PBEWITHSHA1ANDDESEDE
- **Использование:** Java-приложения, Tomcat SSL

#### PKCS#12 (PFX)

- **Стандарт:** RFC 7292
- **Структура:** ASN.1/PKCS#8 AuthenticatedSafe
- **Шифрование:** AES-256 / 3DES
- **Преимущества:** Кросс-платформенный, поддерживается всеми языками

### Генерация ключей

```bash
# RSA 4096-битный ключ
openssl genrsa -out server.key 4096

# Сертификат X.509
openssl req -new -x509 -key server.key -out server.crt -days 365 \
  -subj "/C=RU/ST=Moscow/O=CryptoServices/CN=localhost"

# PKCS#12 контейнер
openssl pkcs12 -export -out keystore.p12 -inkey server.key -in server.crt \
  -passout pass:changeit
```

### Цепочка доверия (Certificate Chain)

```
┌──────────────────────────────────────────────────────────────┐
│                  Цепочка доверия                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  End-Entity Cert (server.crt)                               │
│    ↑ подписан                                                │
│  Intermediate CA Certificate                                │
│    ↑ подписан                                                │
│  Root CA Certificate (Truststore)                           │
│    ↑ самоподписан                                            │
│  Trust Anchor                                               │
│                                                              │
│  Верификация: End-Entity → CA → Root → Trust Anchor        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Безопасное хранение

| Практика              | Описание                                      |
| --------------------- | --------------------------------------------- |
| ENV переменные        | Пароли передаются через окружение, не в коде  |
| Приватные ключи       | Не коммитить в VCS, добавить в `.gitignore`   |
| Права доступа         | Файлы ключей: `chmod 600` (только владелец)   |
| Вращение ключей       | Менять ключи регулярно и при компрометации    |
| Резервное копирование | Шифрованное резервное копирование контейнеров |

## 5. Защита канала (TLS/mTLS)

### TLS handshake (упрощённо)

```
Клиент                                        Сервер
  │                                              │
  │── ClientHello ──────────────────────────────>│  (версии TLS, cipher suites)
  │                                              │
  │<── ServerHello ──────────────────────────────│  (выбранный cipher suite)
  │<── Certificate ──────────────────────────────│  (сертификат сервера)
  │<── ServerKeyExchange ────────────────────────│  (ключи, если нужно)
  │<── ServerHelloDone ──────────────────────────│
  │                                              │
  │── ClientKeyExchange ────────────────────────>│  (pre-master secret)
  │── ChangeCipherSpec ─────────────────────────>│
  │── Finished ─────────────────────────────────>│
  │                                              │
  │<── ChangeCipherSpec ─────────────────────────│
  │<── Finished ─────────────────────────────────│
  │                                              │
  │<═══ Зашифрованный трафик (AES-GCM) ═════════│
```

### mTLS (Mutual TLS)

При mTLS клиент также предоставляет сертификат:

```
Клиент                                        Сервер
  │── ClientHello ──────────────────────────────>│
  │<── ServerHello + Certificate ────────────────│
  │<── CertificateRequest ───────────────────────│  (запросить сертификат клиента)
  │── Certificate ──────────────────────────────>│  (сертификат клиента)
  │── CertificateVerify ────────────────────────>│  (подпись клиента)
  │── ClientKeyExchange ────────────────────────>│
  │...
```

**Конфигурация в проекте:**

- Java: Tomcat с `keystore.p12` + `truststore.jks`
- Go: `server.ListenAndServeTLS(certFile, keyFile)`

## 7. Рекомендации по безопасности

### Криптографические рекомендации

| Рекомендация          | Статус        | Примечание                     |
| --------------------- | ------------- | ------------------------------ |
| RSA ключ ≥ 2048 бит   | ✅ 4096 bit   | 4096 bit = высокий уровень     |
| AES ≥ 128 бит         | ✅ 256 бит    | Максимальная стойкость         |
| SHA-2 ≥ 256 бит       | ✅ SHA-256    | Рекомендован NIST              |
| Уникальный IV для GCM | ✅ Random 12B | Каждый новый при шифровании    |
| HSM для продакшена    | ⚠️            | Использовать аппаратные модули |
| Вращение ключей       | ⚠️            | Добавить политику ротации      |

### Рекомендации по PBKDF2

| Параметр                | Текущее     | Рекомендовано (OWASP) |
| ----------------------- | ----------- | --------------------- |
| Итераций                | 10000       | ≥ 600000 (2023)       |
| Соль                    | 16 байт     | 16 байт (OK)          |
| Псевдослучайная функция | HMAC-SHA256 | OK                    |

### Атаки и защита

| Атака               | Защита                 | Статус                   |
| ------------------- | ---------------------- | ------------------------ |
| Padding oracle      | GCM не уязвим          | ✅                       |
| Replay атаки        | Timestamp в данных     | ⚠️ Реализовать           |
| Brute-force паролей | PBKDF2 с итерациями    | ✅                       |
| Rainbow tables      | Соль в PBKDF2          | ✅                       |
| Side-channel        | Constant-time операции | ⚠️ Зависит от провайдера |

### OWASP Cryptographic Failures (A02:2021)

1. **Не использовать устаревшие алгоритмы:** DES, 3DES, RC4, MD5, SHA-1
2. **Не реализовывать криптографию вручную:** Использовать проверенные библиотеки
3. **Защищать криптографические ключи:** ENV, HSM, KMS
4. **Логировать без утечек:** Не логировать ключи, токены, зашифрованные данные
5. **Аудит:** Регулярно проверять криптографическую конфигурацию
